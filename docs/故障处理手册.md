## 1. 故障排查流程

### 1.1 标准排查步骤

1. **确认故障现象**
   - 记录错误信息
   - 记录发生时间
   - 记录影响范围

2. **检查服务状态**
   ```bash
   # 检查后端服务
   curl http://localhost:8001/health
   
   # 检查进程
   ps aux | grep uvicorn
   
   # 检查端口
   netstat -tlnp | grep 8001
   ```

3. **查看日志**
   ```bash
   cd ai-service
   tail -f logs/app.log
   ```

4. **检查依赖服务**
   - 数据库连接
   - Redis连接
   - AI服务连接

5. **尝试重启服务**

6. **如问题持续，联系技术支持**

### 1.2 紧急程度分级

| 级别 | 描述 | 响应时间 |
|------|------|----------|
| P0 | 系统完全不可用，所有用户受影响 | 立即 |
| P1 | 核心功能不可用，大量用户受影响 | 30分钟内 |
| P2 | 次要功能异常，部分用户受影响 | 2小时内 |
| P3 | 轻微问题，不影响核心功能 | 1个工作日内 |

## 2. 常见问题

### 2.1 服务启动问题

#### 问题1：端口被占用

**现象：**
```
ERROR: [Errno 48] Address already in use
```

**排查：**
```bash
# Windows
netstat -ano | findstr :8001

# Linux
netstat -tlnp | grep 8001
```

**解决：**
```bash
# Windows - 查找并结束进程
taskkill /F /PID <pid>

# Linux
kill -9 <pid>

# 或修改端口（在.env中修改PORT）
```

#### 问题2：依赖缺失

**现象：**
```
ModuleNotFoundError: No module named 'xxx'
```

**排查：**
```bash
cd ai-service
pip list
```

**解决：**
```bash
pip install -r requirements.txt
```

#### 问题3：配置错误

**现象：**
```
KeyError: 'SECRET_KEY'
```

**排查：**
```bash
cd ai-service
cat .env
```

**解决：**
检查`.env`文件，确保所有必需配置项都已设置。参考`.env.example`。

#### 问题1：数据库连接失败

**现象：**
```
sqlalchemy.exc.OperationalError: (sqlite3.OperationalError)
```

**排查：**
```bash
cd ai-service
# 检查数据库文件
ls -la data/
# 检查数据库权限
```

**解决：**
```bash
# 创建数据目录
mkdir -p data
# 初始化数据库
python init_db.py
```

查询返回错误数据或无法读取。

**排查：**
```bash
cd ai-service
python check_db.py
```

```bash
# 从备份恢复
python backup_db.py restore backups/schools_backup_xxx.db
```

#### 问题3：数据库锁

**现象：**
```
database is locked
```

**解决：**
```bash
# 重启服务
# 或检查是否有其他进程正在写入数据库
```

### 2.3 Redis问题

#### 问题1：Redis连接失败

**现象：**
```
redis.exceptions.ConnectionError: Error 111 connecting to localhost:6379.
```

**排查：**
```bash
# 检查Redis是否运行
redis-cli ping

# Windows
tasklist | findstr redis

# Linux
ps aux | grep redis
```

**解决：**
```bash
# 启动Redis
# Windows
redis-server

# Linux
systemctl start redis-server
# 或
service redis start
```

#### 问题2：Redis内存不足

**现象：**
```
OOM command not allowed when used memory > 'maxmemory'
```

**排查：**
```bash
redis-cli info memory
```

**解决：**
```bash
# 清理缓存
redis-cli FLUSHALL

# 或增加Redis内存限制（修改redis.conf）
maxmemory 2gb
maxmemory-policy allkeys-lru
```

### 2.4 API问题

#### 问题1：500 Internal Server Error

**排查：**
1. 查看后端日志：`tail -f logs/app.log`
2. 检查数据库连接
3. 检查Redis连接

**解决：**
根据日志中的具体错误信息进行修复。

#### 问题2：CORS错误

**现象：**
```
Access to XMLHttpRequest has been blocked by CORS policy
```

**排查：**
检查`.env`中的`ALLOWED_ORIGINS`配置。

**解决：**
```bash
# 在.env中设置
ALLOWED_ORIGINS=https://your-domain.com,https://www.your-domain.com
```

#### 问题3：429 Too Many Requests

**现象：**
请求被频率限制拦截。

**排查：**
查看速率限制配置。

**解决：**
- 等待一段时间后重试
- 或调整速率限制配置
- 或使用管理员账号（有更高的限制）

### 2.5 前端问题

#### 问题1：页面白屏

**排查：**
1. 打开浏览器开发者工具（F12）
2. 查看Console是否有错误
3. 查看Network是否有请求失败

**解决：**
- 检查API地址配置
- 清除浏览器缓存
- 检查前端构建是否正确

#### 问题2：静态资源404

**排查：**
检查nginx配置中的静态文件路径。

**解决：**
```nginx
server {
    location / {
        root /path/to/frontend/dist;
        try_files $uri $uri/ /index.html;
    }
}
```

#### 问题3：登录状态丢失

**现象：**
刷新页面后需要重新登录。

**排查：**
检查localStorage中的token是否被清除。

**解决：**
- 检查前端token保存逻辑
- 检查token过期时间设置

### 2.6 AI服务问题

#### 问题1：DeepSeek API调用失败

**现象：**
```
DeepSeek API error: 401 Unauthorized
```

**排查：**
1. 检查API Key是否正确
2. 检查API Key是否有效
3. 检查网络连接

**解决：**
```bash
# 在.env中设置正确的API Key
DEEPSEEK_API_KEY=your-valid-api-key
```

#### 问题2：AI响应超时

**现象：**
```
RequestTimeoutError
```

**排查：**
检查DeepSeek API的响应时间。

**解决：**
```bash
# 增加超时时间（在.env中）
DEEPSEEK_TIMEOUT=60
```

#### 问题3：配额不足

**现象：**
```
Insufficient quota
```

**解决：**
- 检查DeepSeek账户配额
- 考虑升级套餐
- 或实现降级方案（使用缓存或本地规则）

## 3. 性能问题

### 3.1 响应慢

**排查步骤：**
1. 查看Prometheus指标
2. 检查数据库查询性能
3. 检查Redis命中率
4. 查看服务器资源使用情况

**优化建议：**
- 添加数据库索引
- 启用Redis缓存
- 使用CDN加速静态资源
- 增加服务器资源

### 3.2 内存占用高

**排查：**
```bash
# 查看进程内存使用
ps aux --sort=-%mem | head

# Python内存分析
pip install memory-profiler
```

**解决：**
- 优化数据库查询
- 限制缓存大小
- 定期重启服务
- 增加服务器内存

### 3.3 磁盘空间不足

**排查：**
```bash
# Windows
dir

# Linux
df -h
du -sh *
```

**解决：**
```bash
# 清理日志
cd ai-service
find logs -name "*.log" -mtime +30 -delete

# 清理旧备份
find backups -name "*.db" -mtime +7 -delete

# 清理临时文件
rm -rf temp/*
```

## 4. 日志分析

### 4.1 日志位置

```
ai-service/logs/
├── app.log          # 应用日志
├── error.log        # 错误日志
├── access.log       # 访问日志
└── audit.log        # 审计日志
```

### 4.2 日志级别

- `DEBUG` - 调试信息
- `INFO` - 一般信息
- `WARNING` - 警告信息
- `ERROR` - 错误信息
- `CRITICAL` - 严重错误

### 4.3 常用日志查询命令

```bash
# 查看最新日志
tail -f logs/app.log

# 查看错误日志
grep ERROR logs/app.log

# 查看特定时间的日志
grep "2026-05-18 10:" logs/app.log

# 统计错误数量
grep ERROR logs/app.log | wc -l

# 查看最慢的请求
grep "request" logs/app.log | sort -k 5 -r | head
```

## 5. 紧急恢复

### 5.1 服务重启流程

```bash
# 1. 停止服务
pkill -f uvicorn

# 2. 检查服务是否停止
ps aux | grep uvicorn

# 3. 备份当前状态
cp data/schools.db data/schools.db.backup.$(date +%Y%m%d%H%M)

# 4. 启动服务
cd ai-service
python start_service.py

# 5. 验证服务
curl http://localhost:8001/health
```

### 5.2 数据库恢复流程

```bash
# 1. 停止服务
pkill -f uvicorn

# 2. 查看备份列表
cd ai-service
python backup_db.py list

# 3. 选择一个备份恢复
python backup_db.py restore backups/schools_backup_20260518_100000.db

# 4. 启动服务
python start_service.py

# 5. 验证数据
curl http://localhost:8001/api/v1/schools
```

### 5.3 快速回滚（代码版本）

```bash
# 1. 查看git历史
git log --oneline

# 2. 回滚到上一个稳定版本
git reset --hard HEAD~1

# 3. 重启服务
pkill -f uvicorn
cd ai-service
python start_service.py
```

### 5.4 降级方案

当AI服务不可用时，系统可以：
1. 使用缓存的推荐结果
2. 使用本地规则进行推荐
3. 显示友好的提示信息
